Why Prompt Engineering Matters

The quality of AI responses depends heavily on how you ask questions. A well-crafted prompt can mean the difference between getting useless output and a detailed, helpful solution. This mini-course teaches you how to write effective prompts for GitHub Copilot, ChatGPT, Claude, and Google Gemini.

Key Principle: Be specific, provide context, and explain your learning level.

üìö Common Mistakes & How to Fix Them

Scenario: Debugging Code

‚ùå Weak Prompt

"Fix my code"

‚úÖ Strong Prompt

"I'm getting a NullPointerException on line 42 in my Java program for CSC-113. Here's my code:

[paste code here]

The error occurs when I try to call myObject.getName(). I've initialized myObject in the constructor. Can you:
1. Explain why this NullPointerException is happening
2. Show me how to fix it
3. Explain how to prevent this in the future"
Why this works: Specific error, code context, line number, what you've tried, and what you want to learn.

Scenario: Learning New Concepts

‚ùå Weak Prompt

"Explain inheritance"

‚úÖ Strong Prompt

"I'm learning Java inheritance in CSC-114 (Object-Oriented Programming). I understand that classes can extend other classes, but I'm confused about:

1. The difference between 'extends' and 'implements'
2. When to use inheritance vs. composition
3. How method overriding works

Can you explain these concepts using a practical example with Animal as the parent class and Dog/Cat as child classes? I learn best with code examples and visual diagrams."
Why this works: Context (CSC-114), specific confusion points, preferred learning style, and concrete example request.

Scenario: Generating Code

‚ùå Weak Prompt

"Write a function to sort an array"

‚úÖ Strong Prompt

"I need a Java method for my CSC-113 assignment that:

- Takes an ArrayList of integers as input
- Sorts it in ascending order using bubble sort (required by assignment)
- Returns the sorted ArrayList
- Includes comments explaining each step
- Follows Java naming conventions

Please write the method and explain:
1. How bubble sort works
2. The time complexity (Big O notation)
3. When I should use this vs. built-in sort methods"
Why this works: Language specified, exact requirements, learning goals clear, asks for explanation not just code.

Scenario: Design Decisions (CSC-114)

‚ùå Weak Prompt

"How should I structure my program?"

‚úÖ Strong Prompt

"I'm building a library management system for my CSC-114 final project. The system needs to:

- Manage books (add, remove, search)
- Track borrowers
- Handle due dates and fines
- Generate reports

Current approach: I have one big LibrarySystem class with 20+ methods.

Questions:
1. Should I use multiple classes? If so, which ones and why?
2. What design patterns would fit this problem? (We've learned Factory, Observer, Strategy)
3. How should these classes interact?

Please suggest a class structure with UML diagram or text representation, and explain your reasoning using SOLID principles."
Why this works: Project context, current approach, specific design patterns studied, asks for reasoning not just solution.

Scenario: Understanding Errors

‚ùå Weak Prompt

"What does this error mean?"

‚úÖ Strong Prompt

"I'm getting this error in my Java program:

Error: incompatible types: String cannot be converted to int
  at line 23: int age = userInput;

Context:
- I'm reading user input using Scanner in CSC-113
- userInput is declared as: String userInput = scanner.nextLine();
- I need to convert the String to an integer

Questions:
1. Why can't Java automatically convert String to int?
2. What's the correct way to do this conversion?
3. How should I handle cases where the user enters non-numeric input?

I'm a beginner, so please explain like I'm learning this for the first time."
Why this works: Full error message, surrounding code context, specific follow-up questions, learning level stated.

Scenario: Code Review Request

‚ùå Weak Prompt

"Is this code good?"

‚úÖ Strong Prompt

"I've written this Calculator class for CSC-114. Can you review it for:

1. Code quality and best practices
2. Proper use of OOP principles
3. Edge case handling
4. Naming conventions
5. Whether it follows SOLID principles

Here's my code:
[paste code]

Specific concerns:
- I'm not sure if I should make all methods static
- Is my error handling sufficient?
- Should I use inheritance or interfaces for extensibility?

Please point out what's good AND what needs improvement. I want to learn, not just get it working."
Why this works: Specific review criteria, concrete concerns, explicitly asks for learning feedback.

üéØ Tool-Specific Prompting Tips


                

                    

                        
ü§ñ GitHub Copilot

                        
    
                            
  • Use comments to guide: Write detailed comments describing what you want, then let Copilot generate
    • 
                            
  • Context matters: Name variables/methods descriptively - Copilot learns from your naming
    • 
                            
  • Iterate: Accept suggestion, then refine with more comments
    • 
                            
  • Example: // Create a method that validates email addresses using regex and returns true if valid
    • 
                        

                    


                    

                        
üí¨ ChatGPT

                        
    
                            
  • Conversational: Treat it like talking to a tutor - ask follow-ups
    • 
                            
  • Request format: "Explain like I'm 5" or "Give me code examples" or "Show me a diagram"
    • 
                            
  • Iterate on responses: "Can you simplify that?" or "Show me a different approach"
    • 
                            
  • State your level: "I'm a beginner" vs "I'm familiar with OOP but not design patterns"
    • 
                        

                    


                    

                        
üß† Claude

                        
    
                            
  • Upload full projects: Claude's 200K context means you can upload multiple files at once
    • 
                            
  • Ask for deep analysis: "Analyze my entire codebase for design pattern opportunities"
    • 
                            
  • Request structured output: "Give me a step-by-step refactoring plan with before/after code"
    • 
                            
  • Architecture questions: Claude excels at high-level design discussions
    • 
                        

                    


                    

                        
üíé Google Gemini

                        
    
                            
  • Use Deep Research: "Research the best practices for [TOPIC] in 2026"
    • 
                            
  • Documentation questions: "Find and summarize the official [LIBRARY] documentation for [FEATURE]"
    • 
                            
  • Multimodal: Upload screenshots of errors or diagrams for analysis
    • 
                            
  • Large context: Paste entire documentation pages for summarization
    •
