üé≠ Role Focus: Product Owner (Design Phase)

What changes in the design phase?

You're still the Product Owner, but now you're working with how the system will work, not just what it will do. You'll create visual representations of the system architecture, database structure, and user interfaces.

Key Responsibilities This Module:

  • Translate user stories into system architecture diagrams
  • Design database schema to support required features
  • Create wireframes showing user flow through the system
  • Plan Sprint 1: Select stories, define sprint goal, estimate effort
  • Ensure designs align with stakeholder needs from Module 2

Why This Matters: Good design catches problems before coding begins. A well-designed system is easier to build, test, and maintain.

üí° Andrew's Tips for Success

  • Start simple with diagrams: Context diagrams should fit on one page. If your DFD has more than 7 processes, you're going too deep.
  • Wireframes aren't art projects: Low-fidelity is fine. Boxes and labels communicate structure. Save pretty designs for later.
  • Normalize your database: Most systems need at least 3NF (Third Normal Form). If you're storing the same data in multiple tables, rethink your design.
  • Sprint goals aren't lists: "Complete login, dashboard, and reports" isn't a goal. "Enable users to securely access their personalized dashboard" is a goal.
  • Don't overcommit in Sprint 1: You're learning the process. Pick fewer stories than you think you can finish. Better to under-promise and over-deliver.
  • Use design feedback early: Share diagrams with classmates or stakeholders before polishing them. Catching design flaws now saves coding time later.

üì£ Andrew's Note

This is where your project starts to feel real. You're no longer just writing about what you'll build‚Äîyou're designing how it will work. That's both exciting and intimidating. Don't let perfect be the enemy of good. Your first context diagram will probably be wrong in some way. That's fine. Design is iterative.

Sprint 1 is about learning the process, not shipping a perfect product. You're going to make mistakes with estimation. You're going to discover stories that are bigger than you thought. You might not finish everything you planned. All of that is normal and expected. The goal is to practice the sprint cycle: plan, execute, review, adapt.

By the end of Module 3, you should have a clear picture of what you're building and a concrete plan for Sprint 1. That foundation will carry you through the rest of the semester.
