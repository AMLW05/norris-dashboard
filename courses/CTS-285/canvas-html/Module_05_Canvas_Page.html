üé≠ Role Focus: Developer

What does a Developer do in Agile?

Developers build the actual product, but in Agile, they're also responsible for quality, testing, and collaboration. Developers work closely with the Product Owner to understand requirements and with the Scrum Master to remove blockers.

Key Responsibilities This Module:

  • Implement user stories selected for Sprint 2
  • Write test cases and ensure code is testable
  • Conduct code reviews for teammates (or self-review with documented criteria)
  • Maintain development standards (naming conventions, code structure)
  • Update task statuses and communicate progress in standups
  • Balance "getting it done" with "getting it done right"

Why This Matters: Technical debt compounds. Shortcuts taken today become blockers tomorrow. Learning to build quality into the development process‚Äînot bolt it on afterward‚Äîis essential for sustainable software delivery.

üî¨ Hands-On Activity: Code Review Lab (90 minutes)

What you'll experience:

You'll practice code review skills using sample code from a mock project. You'll identify bugs, suggest improvements, assess maintainability, and provide feedback in a constructive format.

What You'll Practice:

  • Reading code for understanding (not just looking for errors)
  • Identifying code smells (duplicated logic, hard-coded values, poor naming)
  • Providing actionable feedback (not just "this is bad")
  • Balancing critique with encouragement
  • Using code review checklists systematically

Review Criteria You'll Apply:

  • Correctness: Does the code do what it's supposed to do?
  • Readability: Can someone else understand this code in 6 months?
  • Testability: Is this code easy to test?
  • Security: Are there obvious vulnerabilities?
  • Performance: Are there obvious inefficiencies?

üí° Andrew's Tips for Success

  • Test as you go, not at the end: Write tests immediately after (or before!) implementing features. End-of-sprint testing is always rushed.
  • Code reviews catch more than bugs: They're knowledge transfer. Document your thinking in code comments so reviewers understand your approach.
  • Definition of Done prevents scope creep: Agree what "done" means before starting (tests written? documentation updated? peer reviewed?).
  • Technical debt is okay‚Äîif tracked: Sometimes you need to ship fast. Document shortcuts you take so you can fix them later.
  • Good commit messages are documentation: "Fixed bug" is useless. "Fixed off-by-one error in pagination (issue #47)" tells a story.
  • Run your own code through the review checklist: Catch your own issues before someone else does. It's faster and less embarrassing.

üì£ Andrew's Note

This is where you feel the tension between shipping features and building quality. In Sprint 2, you'll be tempted to skip tests or cut corners to hit your sprint goal. Resist that urge. The point of this module is to practice sustainable pace‚Äîworking in a way you could maintain for months, not just two weeks.

Code review feels awkward at first. It's uncomfortable to critique your own work or have others critique yours. But it's essential. The best developers I know actively seek code review because they know it makes them better. Embrace the discomfort.

If you finish Sprint 2 without writing a single test, you didn't finish Sprint 2. Testing isn't optional‚Äîit's part of development.
