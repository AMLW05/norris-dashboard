## Course Description

This course introduces established and evolving methodologies for the analysis, design, and development of an information system. Emphasis is placed on system characteristics, managing projects, prototyping, CASE/OOM tools, and systems development life cycle phases. Upon completion, students should be able to analyze a problem and design an appropriate solution using a combination of tools and techniques.

**Our Approach**: You'll learn the SDLC by building a real information system from scratch across 16 weeks, rotating through project roles (Product Owner, Scrum Master, Developer, Release Manager) while practicing Agile/Scrum methodologies that prepare you for CTS-289's capstone collaboration with GRD-242 students.

## Semester-Long Project

You'll build **ONE information system** across 16 weeks, progressing through all SDLC phases.

### Project Options (Choose One)

**Option A: Business System**
- E-commerce platform, inventory management, CRM, scheduling system
- Example: Small business booking system with customer portal

**Option B: Educational/Productivity Tool**
- Learning management tool, task tracker, note-taking app, study organizer
- Example: Collaborative study guide platform

**Option C: Community/Social System**
- Event planning tool, community board, resource sharing, volunteer coordination
- Example: Neighborhood resource exchange platform

**Option D: Propose Your Own**
- Must include: multiple user roles, data persistence, authentication, 15+ user stories

## Success in This Course

### Time Commitment

- **In-class time**: 4 hours/week (2 hrs lecture, 2 hrs lab)
- **Out-of-class work**: 6-8 hours/week average
  - Varies by module (lighter in Module 1, heavier during sprints)
- **Total**: ~10-12 hours/week

### Tips for Success

**1. Start assignments early** - Sprint work can't be crammed into one day

**2. Use the templates** - Planning sheets, standup formats, retrospectives are scaffolding, not busywork

**3. Commit to GitHub frequently** - Small, frequent commits > one giant commit. Shows process and protects your work.

**4. Attend office hours** - Design review before finalizing diagrams saves time

**5. Take retrospectives seriously** - Your Sprint 2 goes smoother when you learn from Sprint 1

**6. Scope ruthlessly** - MVP means Minimum Viable Product. Cut features liberally. You can always add more later.

**7. Ask questions early** - Stuck for 30+ minutes? Ask. Don't suffer in silence.

**8. Read assignment instructions completely** - Deliverables checklist exists for a reason
