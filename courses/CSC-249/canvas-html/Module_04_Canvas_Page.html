Code Preview: Singly Linked List Node

Foundation of all linked structures:

// Node class
class Node {
public:
    int data;
    Node* next;

    Node(int val) : data(val), next(nullptr) {}
};

// LinkedList class
class LinkedList {
private:
    Node* head;

public:
    LinkedList() : head(nullptr) {}

    // Insert at beginning
    void insertFront(int val) {
        Node* newNode = new Node(val);
        newNode->next = head;
        head = newNode;
    }

    // Search for value
    bool search(int val) {
        Node* current = head;
        while (current != nullptr) {
            if (current->data == val) return true;
            current = current->next;
        }
        return false;
    }

    // Display list
    void display() {
        Node* current = head;
        while (current != nullptr) {
            std::cout << current->data << " -> ";
            current = current->next;
        }
        std::cout << "NULL\n";
    }
};

Key insight: Pointers connect nodes into chains. Master pointer manipulation and you unlock all advanced data structures.

Drew's Tips for Success

  • Draw the pointers: Before coding, draw nodes and arrows on paper. Visualizing prevents null pointer crashes.
  • Edge cases matter: Empty list, single node, inserting at head/tail—test all scenarios.
  • Memory leaks are real: In C++, every 'new' needs a 'delete'. Memory management is your responsibility.
  • Sprint planning honesty: Don't overcommit. Better to complete 20 points confidently than fail 30 points.
  • Daily standups build rhythm: Update standup.md every 2-3 days minimum. It keeps you accountable.
  • Burndown chart tells the truth: If you're not making progress by Day 5, ask for help. Don't wait until Day 13.
  • Retrospectives are gold: "I underestimated pointer debugging time" is learning. Apply that to Sprint 2.

Drew's Note

This is where CSC-249 gets real. Sprint 1 combines technical challenge (linked lists) with professional workflow (Scrum). You'll plan work, commit to it publicly, track progress daily, and reflect on what went wrong. This is how software teams operate in industry.

Linked lists break your brain at first. Pointers are invisible, drawing them helps. Inserting at the head is easy. Deleting from the middle is where you'll learn. Segmentation faults teach you to respect memory.

The retrospective is the most important assignment. "What went wrong?" isn't failure—it's data for Sprint 2. Teams that retrospect well improve fast. Teams that don't, repeat mistakes.
