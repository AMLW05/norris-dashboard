Code Preview: Hash Table with Chaining

O(1) average-case lookup:

class HashTable {
private:
    static const int TABLE_SIZE = 10;
    std::vector<std::list<std::pair<std::string, int>>> table;

public:
    HashTable() : table(TABLE_SIZE) {}

    // Hash function
    int hash(const std::string& key) {
        int hashValue = 0;
        for (char c : key) {
            hashValue += c;
        }
        return hashValue % TABLE_SIZE;
    }

    // Insert key-value pair
    void insert(const std::string& key, int value) {
        int index = hash(key);
        // Check if key exists, update if so
        for (auto& pair : table[index]) {
            if (pair.first == key) {
                pair.second = value;
                return;
            }
        }
        // Key doesn't exist, add new pair
        table[index].push_back({key, value});
    }

    // Search for key
    bool search(const std::string& key, int& value) {
        int index = hash(key);
        for (const auto& pair : table[index]) {
            if (pair.first == key) {
                value = pair.second;
                return true;
            }
        }
        return false;
    }
};

Key insight: Hash function maps keys to indices. Chaining handles collisions with linked lists. O(1) average case, O(n) worst case.

Drew's Tips for Success

  • Good hash functions distribute uniformly: Avoid clustering. Test your hash function on real data.
  • Load factor threshold: Rehash when load factor exceeds 0.7 to maintain O(1) performance.
  • Chaining vs open addressing: Chaining is simpler (uses linked lists). Open addressing is faster (no pointers).
  • Velocity-based planning: Sprint 1 velocity is your best predictor. Don't overcommit in Sprint 2.
  • Burndown comparison: Compare Sprint 1 vs Sprint 2 burndown charts. Are you improving?
  • Retrospective insights matter: Apply Sprint 1 lessons to Sprint 2. Did you estimate better? Commit less?

Drew's Note

Hash tables are the secret weapon of data structures. Dictionaries in Python, Maps in C++, Objects in JavaScript—all hash tables under the hood. O(1) lookup feels like magic when you build it yourself and watch it scale to millions of elements.

Velocity tracking is where PM gets real. Sprint 1 gave you baseline data. Sprint 2 tests whether you can plan realistically. Teams that plan using velocity deliver consistently. Teams that ignore it, overcommit and fail.
