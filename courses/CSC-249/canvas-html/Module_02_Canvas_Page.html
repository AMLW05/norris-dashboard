Code Preview: Recursive vs Iterative Factorial

See the difference between recursive and iterative approaches:

// Recursive Factorial
int factorial_recursive(int n) {
    if (n <= 1) return 1;  // Base case
    return n * factorial_recursive(n - 1);  // Recursive case
}

// Iterative Factorial
int factorial_iterative(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i;
    }
    return result;
}

// Benchmark comparison
#include <chrono>
auto start = std::chrono::high_resolution_clock::now();
int result = factorial_recursive(20);
auto end = std::chrono::high_resolution_clock::now();
auto duration = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
std::cout << "Recursive: " << duration.count() << " microseconds\n";

Key insight: Recursive is elegant but has overhead. Iterative is faster but less intuitive for some problems.

Drew's Tips for Success

  • Benchmarking mindset: Don't guess performance—measure it. Run tests on 100, 1000, 10000 elements and graph the results.
  • Big O from experimentation: If doubling input size doubles runtime, it's O(n). If it quadruples, it's O(n²). Theory confirms what you measure.
  • Recursion base cases: Every recursive function needs a stopping condition. Forget it, and you'll stack overflow.
  • Tower of Hanoi is mind-bending: Don't try to trace it in your head—run it, watch it, then understand it.
  • Memoization magic: Fibonacci(40) takes forever recursively. Add memoization and it's instant. That's the power of caching.
  • User stories for algorithms: "As a developer, I want to calculate factorial recursively, so that I understand recursive thinking" is a valid story.

Drew's Note

Big O notation feels like math class, but it's actually an experimentalist's tool. You'll measure algorithms first, then learn the theory confirms what you already observed. If you double the input size and runtime doubles, that's O(n). If it quadruples, that's O(n²). The notation just formalizes what benchmarking teaches you.

Recursion is a superpower. It's how compilers work, how trees traverse, how backtracking algorithms solve mazes. But it's also slow and can blow your stack. You need to understand both when to use it (elegant solutions) and when to avoid it (performance-critical code).

Tower of Hanoi will break your brain the first time. That's normal. Run it with 3 disks, watch it move, then try to explain what happened. Understanding comes from watching, not from reading.
